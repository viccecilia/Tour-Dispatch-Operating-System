TourFlow 多端业务脉络准则

用于旅行社端、车公司端、司机端、平台总控的功能权限、数据库落点、动作联动和异常场景回归验证。每次 Web / 小程序变更、上传或服务器同步后，都按本准则检查主链路是否仍然通顺。

一、核心准则

任何端口的功能修改，都不能破坏这些主约束。
发布基线
单一订单主线

所有业务最终落到 orders。入单、订单大厅、派车、司机执行、结算都围绕同一个订单状态推进。

订单大厅只负责市场动作

auction_listings 只记录发布、竞价、一口价、流拍、撤回。被接单后才暴露双方联系人。

派车只负责执行动作

assignments 记录司机和车辆；司机端只看自己的任务，旅行社只能看跟踪状态和必要执行信息。

租户隔离优先

旅行社按 agency_id 隔离；车公司按 tenant_id 隔离。公开大厅展示脱敏信息，不能越权读取。

倒拍规则

起拍价是放出价格，车公司每次竞拍向下减价；一口价是快速成交价，允许低于起拍价。

变更必须通知

已接单、已派车或执行中的订单变更后，必须写入记录，并通知车公司、司机、旅行社相关角色。

三、数据库关系示意图

订单是中心，其他表只记录某一类动作或证据。

orders

订单主表：订单号、来源、旅行社、车公司、时间、路线、车型、价格、派车状态、结算状态。

auction_listings

订单大厅发布、起拍价、当前价、一口价、拍卖时长、竞价/接单状态。

assignments

派车结果：司机、车辆、确认、执行、取消、回执。

drivers / vehicles

司机和车辆资源池。车型必须和订单需求匹配：A-3, A-4, H。

notifications

新订单、司机派单、订单变更、取消、结算、异常提醒。

driver_reports

司机确认、出发、到达、完成、异常节点汇报。

location_logs

司机/车辆位置点，支撑旅行社和车公司地图跟踪。

driver_expense_reports

垫付报销、代收上交、金额、备注、状态。

driver_evidence_uploads

领收书、回执、照片证据。

agency profile / payment

旅行社藤本 PDF、担当、电话、打款账号、付款回执。

audit / change logs

谁改了什么、改前改后、通知是否发出、是否需要确认。

四、订单号与可见范围

订单号要能追责，但不能把内部信息随便暴露给大厅。
内部订单号规则

AGA260615A1-0002：AGA 旅行社，2026-06-15，A1 账号发布，当日第 2 单。

AGA260615A1-0002-SKR260615B1-0003：被 SKR 车公司 B1 账号在 2026-06-15 接单，是 SKR 当日第 3 单。

AGA260615A1-0002-SKR260615B1-0003-KEN6637H：派给 Ken，车辆 6637，车型 H。

不同端口如何暴露

公开订单大厅 不显示完整订单号、不显示旅行社、不显示联系人；可使用短流水或不显示编号。

旅行社/车公司接单后 双方可见完整业务号、担当联系方式和执行信息。

司机端 显示执行号和必要客户/导游信息，不展示竞价来源和车公司利润。

平台端 可查看完整链路号用于审计。

五、动作线 A：旅行社发布，车公司一口价接单

这是订单大厅最关键的联动链路。
旅行社录入订单

旅行社 Web / 小程序录入接机、送机、包车。系统写入 orders，状态为未发布；包车可上传 PDF，机场接送默认不强制 PDF。

旅行社订单列表旅行社日历平台可审计
旅行社发布到订单大厅

写入或更新 auction_listings，订单状态变为发布中。公开大厅只显示日期时间、起终点/主要行程、车型、人数/行李数、起拍价、一口价、剩余时间。

不显示订单号不显示旅行社不显示联系人
车公司管理端一口价接单

车公司管理账号点击一口价，auction_listings.status 变为已接单，orders.dispatch_status 变为已接单/待派车。订单号追加车公司接单段。

旅行社看到车公司担当车公司看到旅行社担当双方可见完整订单内容
联动通知

生成通知：旅行社管理/客服、车公司管理/调度、平台记录。财务端出现待结算占位，但未完成前不请求付款。

notificationsagency settlement pendingcarrier upstream receivable pending

六、动作线 B：车公司派车，司机确认并执行

派车后，司机端、旅行社跟踪和车公司调度必须同步。
调度选择司机和车辆

车公司调度/管理端选择司机和车辆。系统必须校验车型：订单要求 A-3 只能配 A-3，A-4 只能配 A-4，H 只能配 H。强制例外必须由管理账号写原因。

assignmentsdriversvehicles
写入派车记录并追加订单号

创建或更新 assignments；订单状态变为已派车/待司机确认；订单号追加司机和车辆段，例如 -KEN6637H。

车公司派车池更新旅行社跟踪可见司机车辆司机首页出现待确认
司机确认派单

司机小程序首页显示待确认派单。司机确认后，任务进入任务页；首页待确认数减少。通知应该发给司机本人，不应该只出现在调度平台。

assignment confirmeddriver notification read/unread
司机出库与执行

司机未出库时，任务页顶部始终显示点呼出库项。完成车辆、证件、睡眠、酒精等确认后点击出库，开始按顺序执行今日任务。

driver_workflow_eventsdriver_reports
位置与节点同步

司机上报当前位置、出发、到达、完成、异常。旅行社跟踪、车公司地图、平台审计同步显示。

location_logsagency trackingcarrier map
全部任务完成后入库

无后续任务时进入入库流程：入库时间、休息时间、车辆状态、遗失物、异常申告。提交后今天工作结束。

returnedworkday closed

七、动作线 C：订单完成与上下游财务

财务不是一个状态，而是旅行社付款、车公司确认、司机工资/报销/代收三条线。
旅行社 -> 车公司

订单完成后，车公司发起付款请求；旅行社打款并上传回执；车公司确认收款；订单结算完成。

agency_payment_requested receipt_uploaded paid_confirmed
车公司 -> 司机工资

财务按月统计司机接送机数量、包车数量，乘以对应单价，生成司机工资单。

transfer_count charter_count unit_price
司机垫付 / 代收

司机提交金额、备注、领收书照片。垫付是公司报销给司机；代收是司机上交给公司。填错可退回修改。

submitted needs_correction approved

十一、建议补充的自动化检查

后续可以把这部分做成一键验收脚本。
1. 订单大厅全链路脚本
  • AGA 发布接机订单，SKR 一口价接单，SKR 派 Ken + H 车，司机确认，司机出库，完成，车公司发起收款，AGA 上传回执，SKR 确认收款。
  • 断言 orders, auction_listings, assignments, notifications, driver_reports, agency_settlement_status 全部一致。
2. 多租户边界脚本
  • AGA/AGB/AGC 各发 1 单；SKR/KEX/TYO 各接 1 单；检查互相不可见的字段。
  • 公开大厅可以看基础行程和价格，但不能看旅行社、联系人、完整订单号。
3. 异常场景脚本
  • 发布后撤回、超时流拍、已接单后变更、已接单后强制取消、司机费用退回修改、完成未支付逾期。
  • 每个异常都要检查状态、通知、财务、历史记录。
