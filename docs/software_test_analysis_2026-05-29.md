## 2. 发现的问题

| 等级 | 问题 | 结果 | 建议 |
|---|---|---|---|
| P1 | `verify_parser_api.py` / `verify_live_map_runtime.py` 调用 `init_db()` 时触发 `sqlite3.IntegrityError: UNIQUE constraint failed: orders.oid` | 失败 | 检查 `refresh_order_oids()` 的重算顺序和冲突处理；不能在重算时直接撞唯一索引 |
| P2 | `verify_batch_parser_api.py` 未带登录 token 调 `/api/parser/text` | 401 | 脚本应先登录并带 `Authorization`，或明确改成只测匿名接口 |
| P2 | `admin-trial` DNS 仍可能有本机/运营商缓存 | 外部公共 DNS 已正确，本机可能旧 IP | 等待缓存，或本机执行 `ipconfig /flushdns` |
| P3 | Web 构建提示主 JS chunk 超过 500 kB | 不阻断 | 后续高订单量前可做代码分包 |

## 7. 结论

当前系统的核心 API 和主要业务联动已经具备 trial 测试条件。后台 Web 构建、角色权限、账号管理、派车、司机执行、费用、日历、出勤、资源提醒均通过自动化验证。

上线真机测试前建议优先处理：
1. 修复 `refresh_order_oids()` 在重算订单号时可能触发唯一索引冲突的问题。
2. 修复 `verify_batch_parser_api.py` 的登录 token，让批量解析链路纳入自动化。
3. 用微信真机逐项复验：登录/微信绑定免登录、司机确认按钮、tab 跳转一致性、司机资料文件上传。
4. DNS 缓存完全刷新后，再直接用 `https://admin-trial.taxi-airport.jp` 做 Web 手点验收。
