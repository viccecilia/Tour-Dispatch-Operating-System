# R068 真机试运行反馈手册

## 目标

把 Tour Dispatch Operating System 从“功能完整”推进到“真实可用”。本轮不追求新增功能，重点是连续使用、记录问题、形成优化队列。

## 试运行对象

- 调度端 Web Console
- 调度小程序
- 司机小程序
- 后端 Runtime
- 财务台账
- 地图和位置上报

## TASK-001 Dispatcher Real Usage

调度员连续操作至少 1 个完整工作日，建议覆盖：

1. 粘贴 20-50 条真实订单文本。
2. 批量解析。
3. 修改 5 条解析有偏差的订单。
4. 批量确认入库。
5. 按包车 / 接机 / 送机拆分查看订单。
6. 选择 4-8 条订单进行派车。
7. 检查司机确认回执。
8. 打开地图查看司机位置。
9. 打开财务页确认完成订单和费用状态。

记录重点：

- 哪一步找不到入口。
- 哪一步点击次数太多。
- 哪个字段经常需要手改。
- 哪个页面信息太多或太少。

## TASK-002 Driver Real Usage

司机端至少模拟 1 个完整出车日：

1. 首页查看待确认订单。
2. 一次性确认接单。
3. 出库前检查。
4. 点呼出库。
5. 打开任务页查看当天订单。
6. 打开地图页查看当前任务。
7. 到达上车点。
8. 上传或跳过照片节点。
9. 接到客人。
10. 到达终点。
11. 完成订单。
12. 所有订单结束后入库。
13. 提交垫付 / 代收费用。
14. 查看我的页面和工作履历。

记录重点：

- 司机是否 3 秒内知道下一步。
- 是否有“不知道该点哪里”的时刻。
- 出库和任务流程是否混在一起。
- 地图页是否像任务工具。
- 费用报备是否顺手。

## TASK-003 Weak Network Runtime

建议用微信开发者工具或手机网络切换测试：

1. 正常网络下打开司机端。
2. 进入任务地图页。
3. 切换弱网或断网。
4. 点击状态按钮。
5. 上传费用或照片。
6. 恢复网络。
7. 检查是否能继续操作。

记录重点：

- 是否白屏。
- 是否没有提示。
- 是否重复提交。
- 是否状态丢失。
- 是否需要手动重试入口。

## TASK-004 Runtime Friction Analysis

每个问题按下面格式记录：

```text
时间：
角色：调度 / 司机 / 财务 / 管理员
页面：
操作：
预期：
实际：
严重程度：P0 / P1 / P2 / P3
类型：卡顿 / 不顺手 / 找不到 / 多余步骤 / 数据不对 / 文案不清楚
截图：
备注：
```

严重程度定义：

- P0：阻断试运行，必须当天修。
- P1：影响核心流程，进入本周修复。
- P2：影响体验，不阻断使用。
- P3：优化建议。

## TASK-005 Pilot Improvement Queue

优化队列字段：

- id
- priority
- role
- area
- issue
- expected
- evidence
- owner
- status
- next_action

状态：

- new
- triaged
- in_progress
- fixed
- verified
- deferred

## 每日收口

每天试运行结束后做 15 分钟收口：

1. 汇总 P0/P1。
2. 决定第二天是否继续试运行。
3. 把问题写入 `runtime/pilot_feedback/R068/feedback_items.json`。
4. 运行：

```bash
python scripts/generate_pilot_feedback_queue.py
```

5. 查看：

```text
runtime/pilot_feedback/R068/PILOT_IMPROVEMENT_QUEUE.md
```
